# NebariApp tutorial

This page walks you through deploying a `NebariApp` on a local Kind cluster from scratch. By the end you'll have a running app reachable at a real hostname, with HTTPS and (optionally) Keycloak login, all wired up by the [Nebari Operator](https://github.com/nebari-dev/nebari-operator).

{/* This walkthrough mirrors the operator's own quickstart at
    https://github.com/nebari-dev/nebari-operator/blob/main/docs/quickstart.md.
    If that page drifts, re-sync this one. */}

## Prerequisites

You'll need:

- **Docker:** the container runtime Kind uses to host the cluster.
- **`kubectl`:** the command-line client for talking to a Kubernetes cluster.
- **[Kind](https://kind.sigs.k8s.io/):** runs a Kubernetes cluster inside Docker, for local dev.
- **[Helm](https://helm.sh/):** package manager `make setup` uses to install Envoy Gateway and cert-manager.

Clone the [Nebari Operator](https://github.com/nebari-dev/nebari-operator) and run every step from the repository root:

```bash
git clone https://github.com/nebari-dev/nebari-operator.git
cd nebari-operator
```

## 1. Set up the dev cluster

Before installing the operator, you need a cluster for it to run on. This step uses the operator repo's prebuilt scripts to bring one up on your laptop.

A single command creates the Kind cluster and installs everything the operator depends on:

```bash
cd dev
make setup
```

After this finishes, your local cluster can:

- Run containers (via the Kind cluster `nebari-operator-dev`)
- Hand out IP addresses to services (MetalLB)
- Route traffic by hostname (Envoy Gateway, with a shared `nebari-gateway`)
- Issue HTTPS certificates automatically (cert-manager, with a wildcard cert for `*.nebari.local`)
- Handle user sign-in (Keycloak, optional)

Everything else later in this tutorial uses these pieces.

Verify the environment is ready:

```bash
make status
```

You should see a Gateway with an address, confirming the load balancer is up.

## 2. Install the operator

Now that the cluster is ready, install the Nebari Operator into it. Once running, it will watch for any `NebariApp` resources you create and wire them into the cluster automatically.

Build the operator image, load it into the Kind cluster, and deploy:

```bash
# Return to root directory
cd ..

# Build the operator image
make docker-build IMG=quay.io/nebari/nebari-operator:dev

# Load image into Kind cluster
kind load docker-image quay.io/nebari/nebari-operator:dev --name nebari-operator-dev

# Install CRDs
make install

# Deploy operator to cluster
make deploy IMG=quay.io/nebari/nebari-operator:dev
```

Verify the operator is running:

```bash
kubectl get pods -n nebari-operator-system
```

```bash title="Output"
NAME                                                  READY   STATUS    RESTARTS   AGE
nebari-operator-controller-manager-66d9f7fcdf-rlbqc   1/1     Running   0          35s
```

## 3. Deploy a sample app

Before creating a `NebariApp`, you need a running app for it to point at. This step deploys a tiny "hello world" container that returns a fixed message when you visit it. In a real pack, this would be your team's actual application.

First, create a namespace for the app, then add a label (`nebari.dev/managed=true`) that tells the operator it's allowed to act on resources in this namespace:

```bash
kubectl create namespace demo
kubectl label namespace demo nebari.dev/managed=true
```

Then spin up the container and give it an in-cluster address so other workloads can talk to it. The container runs a minimal HTTP server that replies with "Hello from Nebari Operator!" to any request:

```bash
kubectl apply -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello-app
  namespace: demo
spec:
  replicas: 1
  selector:
    matchLabels:
      app: hello
  template:
    metadata:
      labels:
        app: hello
    spec:
      containers:
      - name: hello
        image: hashicorp/http-echo
        args:
          - "-text=Hello from Nebari Operator!"
        ports:
        - containerPort: 5678
---
apiVersion: v1
kind: Service
metadata:
  name: hello-service
  namespace: demo
spec:
  selector:
    app: hello
  ports:
  - port: 80
    targetPort: 5678
EOF
```

At this point, the app is running, but nothing on the cluster routes traffic to it yet.

## 4. Create a `NebariApp`

The `NebariApp` is what connects the hostname to the Service. Apply this manifest to create one:

```bash
kubectl apply -f - <<EOF
apiVersion: reconcilers.nebari.dev/v1
kind: NebariApp
metadata:
  name: hello-app
  namespace: demo
spec:
  hostname: hello-app.nebari.local
  service:
    name: hello-service
    port: 80
  routing:
    tls:
      enabled: true
EOF
```

The operator picks it up and creates:

- An **`HTTPRoute`** that sends traffic for `hello-app.nebari.local` to `hello-service:80`
- A **`Certificate`** for `hello-app.nebari.local` so visitors connect over HTTPS

Verify the `NebariApp` is `Ready`:

```bash
kubectl describe nebariapp hello-app -n demo
```

In the `Conditions` block at the bottom, find the `Ready` condition. As long as its `Status` is `True`, the operator finished setting up the app.

## 5. Test it

:::note macOS one-time setup

On macOS, Docker Desktop runs the cluster in a VM whose network isn't reachable from your host. Install [`docker-mac-net-connect`](https://github.com/chipmk/docker-mac-net-connect) once to add the missing routes:

```bash
brew install chipmk/tap/docker-mac-net-connect
sudo brew services start chipmk/tap/docker-mac-net-connect
```

Once installed, you can reach the cluster's hostnames directly from your browser and terminal.

:::

First, tell your laptop where to find the app by adding it to `/etc/hosts`:

```bash
./dev/scripts/networking/update-hosts.sh hello-app
```

Then use `curl` to send a request to the app:

```bash
curl -k https://hello-app.nebari.local
```

The response should be:

```
Hello from Nebari Operator!
```

## 6. Add login (optional)

By default the app is reachable without any login. If you want only signed-in users to be able to reach it, this step adds a `Keycloak` auth requirement to the `NebariApp`.

Auth enforcement needs a Keycloak realm and admin credentials that the operator can use to provision OAuth clients. Run the dev setup script once to create them:

```bash
./dev/scripts/services/keycloak/setup.sh
```

Then add an `auth` block to the `NebariApp`:

```bash
kubectl apply -f - <<EOF
apiVersion: reconcilers.nebari.dev/v1
kind: NebariApp
metadata:
  name: hello-app
  namespace: demo
spec:
  hostname: hello-app.nebari.local
  service:
    name: hello-service
    port: 80
  routing:
    tls:
      enabled: true
  auth:
    enabled: true
    provider: keycloak
    provisionClient: true
    scopes:
      - openid
      - profile
      - email
EOF
```

The operator sets up a Keycloak login for the app. Open the app's URL in your browser:

```bash
open https://hello-app.nebari.local
```

You should be redirected to the Keycloak login page. The default dev credentials are `admin` / `nebari-admin`.

## 7. Clean up

Delete the `NebariApp` so the operator can clean up the resources it created:

```bash
kubectl delete nebariapp hello-app -n demo
```

Remove the demo namespace, which deletes the sample app's Deployment and Service:

```bash
kubectl delete namespace demo
```

Finally, tear down the dev cluster itself:

```bash
cd dev
make teardown
```

## What you've seen

You've now seen what a `NebariApp` does in practice: declare the hostname, the service, and whether to require login, and the operator wires everything else automatically. A real pack is just this same pattern packaged as a reusable Helm chart.
