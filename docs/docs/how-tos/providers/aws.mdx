This guide walks through deploying NKP on AWS using `nic`, the Nebari Infrastructure Core CLI. It targets operators who own day-to-day cluster operations and need to understand the IAM and cost realities before provisioning.

## What your team gets

When `nic deploy` finishes, your team will have:

- **A managed Kubernetes cluster** ready for workloads (AWS EKS, multi-AZ by default).
- **Automatic TLS** for every service you publish ([cert-manager](https://cert-manager.io/) + Let's Encrypt).
- **Single sign-on** across all services ([Keycloak](https://www.keycloak.org/)).
- **Ingress routing** for any service you expose ([Envoy Gateway](https://gateway.envoyproxy.io/)).
- **Shared storage** that pods on any node can mount ([EFS](https://aws.amazon.com/efs/)).
- **GitOps-driven updates:** future changes flow through a Git repo you own ([ArgoCD](https://argo-cd.readthedocs.io/)).
- **OpenTelemetry** collection ready to plug into your observability backend.

## Prerequisites

### AWS account

You'll need access to an AWS account you can deploy into. If you don't have one, [sign up for AWS](https://aws.amazon.com/free/).

- **Region:** pick one where [EKS is available](https://docs.aws.amazon.com/eks/latest/userguide/eks-regions.html) and has at least two [availability zones](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/using-regions-availability-zones.html). The example uses `us-west-2a` and `us-west-2b`.
- **Quotas:** confirm your region has enough [EKS service quota](https://docs.aws.amazon.com/eks/latest/userguide/service-quotas.html) for the cluster you're about to create. New accounts often need an increase.
- **Cost:** NKP does not fit within the free tier; EKS, NAT gateways, and EFS all bill from day one.

### IAM permissions

Running `nic deploy` needs IAM permissions across these services:

- **Base:** STS, S3 (state bucket), EC2 core, EKS.
- **VPC:** EC2 VPC/subnet/NAT/route-table/security-group/VPC-endpoint actions.
- **IAM:** role create/delete/attach/PassRole.
- **EFS:** file-system and mount-target actions.

For a first deployment, admin credentials are the fastest path. For production, deploy once with admin, then use [IAM Access Analyzer's policy generation](https://docs.aws.amazon.com/IAM/latest/UserGuide/access-analyzer-policy-generation.html) (or CloudTrail) to derive a least-privilege policy from the API calls `nic` actually made. NIC does not ship a pre-built policy today.

### Install nic

The `nic` CLI provisions the Kubernetes cluster and bootstraps the foundational software (ArgoCD, Envoy Gateway, Keycloak, cert-manager). Install it from source:

```bash
git clone https://github.com/nebari-dev/nebari-infrastructure-core.git
cd nebari-infrastructure-core
make install
```

### GitOps repository

NIC uses GitOps: it commits ArgoCD app manifests to a Git repo you own and lets ArgoCD sync them into the cluster.

You'll need:

1. **An empty Git repo on any host reachable from the cluster** (GitHub, GitLab, Bitbucket, self-hosted Gitea, etc.).
2. **Two GitHub personal access tokens** scoped to the GitOps repo. For each, go to [github.com/settings/tokens?type=beta](https://github.com/settings/tokens?type=beta), choose **Only select repositories** and pick your GitOps repo, then generate. The two tokens differ only in the **Contents** permission:
   - **`GIT_TOKEN`** (used by `nic`): set **Contents: read+write**.
   - **`ARGOCD_GIT_TOKEN`** (used by ArgoCD in the cluster): set **Contents: read-only**.

### Secrets and credentials

NIC reads secrets from a `.env` file in the directory you run `nic` from (loaded via [godotenv](https://github.com/joho/godotenv)). From the `nebari-infrastructure-core` clone you made above, copy the template into your working directory:

```bash
cp /path/to/nebari-infrastructure-core/.env.example .env
```

Then fill in the values you need.

Uncomment and fill in at least the AWS credentials and **both** GitOps tokens. NIC needs two separate GitHub personal access tokens for the GitOps repo:

- **`GIT_TOKEN`** (read+write): `nic` uses this from your machine to write ArgoCD application manifests into the repo during deploy.
- **`ARGOCD_GIT_TOKEN`** (read-only): ArgoCD running inside the cluster uses this to continuously pull and apply the foundational-software manifests, keeping the cluster's state in sync with the repo.

Using two scoped tokens means the cluster-side credential only has read access, limiting blast radius if the cluster is compromised.

```bash
# AWS credentials (pick one pattern)
AWS_PROFILE=my-sso-profile          # SSO / IAM Identity Center
# AWS_ACCESS_KEY_ID=...             # OR static keys
# AWS_SECRET_ACCESS_KEY=...

# GitOps repo access (two tokens)
GIT_TOKEN=github_pat_...             # read+write, used by nic during deploy
ARGOCD_GIT_TOKEN=github_pat_...      # read-only, used by ArgoCD to pull manifests
```

## Cost considerations

A NKP deployment provisions several AWS services that bill from day one. Check the AWS pricing pages for current rates in your region.

- **[EKS control plane](https://aws.amazon.com/eks/pricing/):** flat per-cluster hourly rate.
- **[EC2 instances](https://aws.amazon.com/ec2/pricing/on-demand/)** in your node groups: instance hours per running node.
- **[EBS root volumes](https://aws.amazon.com/ebs/pricing/)** on each node: per-GB-month for node disks.
- **[NAT gateways](https://aws.amazon.com/vpc/pricing/):** one per AZ. Hourly rate per gateway plus per-GB data processing.
- **[NLBs](https://aws.amazon.com/elasticloadbalancing/pricing/):** the AWS Load Balancer Controller creates one for ingress. Hourly rate plus Load Balancer Capacity Units (LCU).
- **[EFS](https://aws.amazon.com/efs/pricing/):** per-GB-month for shared cluster storage. `bursting` is the default and is cheap at low utilization.

Configuring a single AZ (`availability_zones: [us-west-2a]`) halves your NAT-gateway bill, but the entire cluster goes down if that AZ has an outage. Reasonable for short-lived test deploys; not for production.

## Configuration

Copy a starter config from the `nebari-infrastructure-core` clone you made above. Pick the one that matches your DNS setup:

- **`aws-config.yaml`** for any DNS provider: you'll create A/CNAME records manually at deploy time using values `nic` prints.
- **`aws-config-with-dns.yaml`** for Cloudflare-hosted domains: `nic` creates the records automatically.

```bash
cp /path/to/nebari-infrastructure-core/examples/aws-config.yaml aws-config.yaml
# or, for the Cloudflare variant:
cp /path/to/nebari-infrastructure-core/examples/aws-config-with-dns.yaml aws-config.yaml
```

:::note
In later steps, `<config-file>` refers to the local copy you just created (e.g., `aws-config.yaml`).
:::

At minimum, edit these fields:

- `project_name`: lowercase alphanumeric (with `-` and `_`), under ~30 chars; names the S3 state bucket, EKS cluster, and tagged resources.
- `domain`: a hostname you own.
- `git_repository.url`: the URL of the empty GitOps repo you created above.
- `git_repository.auth.token_env`: set to `GIT_TOKEN` to use the PAT path you set up in `.env`.
- `cluster.aws.region` and `availability_zones`: pick a region with EKS support and at least two AZs.

For the full schema (brownfield VPC adoption, custom KMS keys, advanced node-group options, Longhorn, log types), see the [NIC configuration reference](https://github.com/nebari-dev/nebari-infrastructure-core/blob/main/docs/design-doc/appendix/16-configuration-reference.md).

## Deploy

From the directory containing your config file:

```bash
# Quick syntax and shape check; no AWS calls.
nic validate -f <config-file>

# Preview what will change without touching AWS.
nic deploy -f <config-file> --dry-run

# Actually provision.
nic deploy -f <config-file>
```

Allow at least 30 minutes for the first deployment. The bulk of the time is EKS control-plane creation (about 10 minutes) and ArgoCD syncing foundational services after the cluster comes up.

If you need to extend the default timeout (large clusters, slow regions), pass `--timeout 1h`.

**After deploy**, DNS handling depends on which config you chose:

- **`aws-config.yaml`:** `nic` prints an A/CNAME record at the end of deploy. Create it at your DNS provider before the cluster becomes reachable.
- **`aws-config-with-dns.yaml`:** `nic` creates the DNS records automatically.

## IAM roles `nic` creates in your account

`nic deploy` creates a few IAM roles in your account so the cluster, the nodes, and the in-cluster controllers can each do their AWS work safely:

| Role | Purpose |
|------|---------|
| EKS cluster role | Lets EKS create the network interfaces, security groups, and log groups the cluster needs. |
| EKS node-group role | Grants nodes the standard EKS worker, ECR-read-only, and CNI policies. |
| AWS Load Balancer Controller role | Lets the in-cluster controller create and manage load balancers for ingress. |
| EFS CSI driver role | Lets the in-cluster driver mount EFS into pods. |

## Verify

:::note
The steps below use [`kubectl`](https://kubernetes.io/docs/tasks/tools/) and the [AWS CLI](https://aws.amazon.com/cli/). Install them if you don't have them.
:::

After `nic deploy` returns, point your kubectl at the cluster and check it:

```bash
aws eks update-kubeconfig --name <project-name> --region <region>

kubectl get nodes
kubectl get pods -A
kubectl get applications -n argocd
```

Substitute `<project-name>` and `<region>` with the values from your config.

All ArgoCD applications should reach `Synced` and `Healthy` within a few minutes. Then:

1. Visit `https://<your-domain>` (the value of `domain:` in your config) to confirm DNS, TLS, and ingress are working. You should land on the Launchpad.
2. Sign in with the Keycloak root credentials `nic` printed at the end of deploy to confirm SSO is working.

:::tip
For an interactive view of all cluster resources (especially handy while watching ArgoCD apps sync), install [k9s](https://k9scli.io/) and run it after the `update-kubeconfig` step.
:::

## Update an existing deployment

To change something about a running cluster (scale a node group, add a `gpu` group, change tags, switch EFS throughput), edit your config and re-run:

```bash
nic deploy -f <config-file> --dry-run    # preview the diff
nic deploy -f <config-file>              # apply it
```

`nic` is declarative, so only the diff is applied.

:::warning
Changing `region`, `project_name`, or `vpc_cidr_block` triggers destructive resource recreation. Treat these as one-way decisions.
:::

## Destroy

When you're done with the cluster (testing complete, project ended, or you want to start fresh), tear it down with `nic destroy`. Run a dry-run first to see what will be removed:

```bash
# Preview what will be destroyed.
nic destroy -f <config-file> --dry-run

# Tear down everything nic created.
nic destroy -f <config-file>
```

`nic destroy` removes EKS, node groups, EFS, VPC components, and the state bucket. If a resource fails to delete (commonly, a leftover load balancer from the cluster's ingress), remove it manually in the AWS console and retry with `--force`:

```bash
nic destroy -f aws-config.yaml --force
```

Always confirm in the AWS console that no orphan resources remain. NAT gateways, load balancers, and EBS volumes can keep billing if they're not cleaned up.
