# Pack architecture

A **Software Pack** is a self-contained, installable artifact published to a GitHub repository. Once installed on NKP, it delivers a Capability such as a chat assistant, BI dashboards, or a model-serving stack.

Each pack is already wired into the platform's login, routing, and TLS, so users can open it from the landing page and start working.

## Why packs exist

A platform that ships everything in one bundle is heavy to install, expensive to run, and slow to change. Packs solve this by giving NKP four properties:

- **Composable:** install only the Capabilities your users need. A team using Superset doesn't have to install LLM Serving, and vice versa.
- **Independent upgrades:** each pack upgrades on its own. Pulling in a new version of one pack leaves every other pack untouched.
- **Bring your own pack:** if no existing pack does what you need, you can build one yourself using the same template and resources OpenTeams uses.
- **Failures don't spread:** if one pack breaks, the other packs keep working. You can debug, roll back, or remove the broken one without touching anything else.

## What a pack contains

A Software Pack is two things glued together:

- **An ArgoCD `Application`** that points at the pack's GitHub repository. ArgoCD reads it and installs the pack's workloads.
- **A `NebariApp` resource** that tells the [Nebari Operator](https://github.com/nebari-dev/nebari-operator) to connect the pack to login, routing, and TLS automatically.

## The NebariApp resource

A `NebariApp` is a short file the pack author writes. It names a hostname, a service to expose, and whether to require login:

```yaml
apiVersion: reconcilers.nebari.dev/v1
kind: NebariApp
metadata:
  name: hello-app
  namespace: demo
spec:
  # public URL for the pack
  hostname: hello.nebari.local

  # workload to send traffic to
  service:
    name: hello-service
    port: 80

  # HTTPS via cert-manager
  routing:
    tls:
      enabled: true

  # single sign-on via Keycloak
  auth:
    enabled: true
    provider: keycloak
    provisionClient: true
```

From this, the Nebari Operator creates:

- An **`HTTPRoute`** that sends traffic for `hostname` to the `service`, so the pack is reachable from the outside.
- A **`Certificate`** for `hostname`, so visitors connect over HTTPS instead of plain HTTP.
- A **Keycloak login** for the pack, so visitors sign in once and use the same account everywhere on the platform.

In short, the pack author writes only the `NebariApp`; the operator handles the rest.

For the full list of `NebariApp` fields, see the [operator configuration reference](https://github.com/nebari-dev/nebari-operator/blob/main/docs/configuration-reference.md).

## Pack lifecycle

A pack takes four steps to get from a GitHub repo to the landing page:

1. **The pack is installed onto a cluster via ArgoCD.** An ArgoCD `Application` manifest points at the pack's repo, and ArgoCD syncs from there.
2. **ArgoCD deploys the pack's workloads and the `NebariApp` into the cluster.**
3. **The Nebari Operator picks up the `NebariApp`** and creates the routing, certificate, and login that the pack needs.
4. **The pack appears on the [landing page](https://github.com/nebari-dev/nebari-landing)** as an available Capability.

## Where packs come from

Packs are published to GitHub repositories, and they come from two channels:

- **Official packs.** Maintained by OpenTeams and published under the [`nebari-dev`](https://github.com/nebari-dev) GitHub organization. See the [pack catalog](/docs/software-packs/) for the current list.
- **Community packs.** Anyone outside OpenTeams can build one, whether that's an open-source maintainer or your own team. A community pack follows the same structure as an official pack. See [Build your own pack](/docs/software-packs/build-your-own) to author one.
